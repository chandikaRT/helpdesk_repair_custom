1 · Executive Summary

The Helpdesk module has been significantly extended via Odoo Studio to implement a full repair & warranty ticket management workflow. The customizations span four models and introduce complex multi-step repair lifecycle logic entirely within Studio — covering receipt intake, serial number tracking, two repair pathways (RUG/warranty vs. normal repair), factory routing, estimation, FSM task integration, cancellation, and document generation.

Core workflow: A helpdesk ticket is the central repair job record. It drives stock transfers (receipts/returns), FSM field-service tasks, sale orders for billing, and stage-based approval gates — all orchestrated through Studio automation, computed fields, and header buttons.

helpdesk.ticket — Form View (id: 4012)

The most complex view customization. Key changes:

  • Ticket name field: forced readonly with placeholder "New"; auto-named by sequence on create
  • Stat button "Repair Trans." added: shows count of stock pickings created from this ticket
  • Readonly conditions on team_id, user_id, partner_id, ticket_type_id — locked when RUG approved or request sent
  • Fields added after ticket_type_id: rug_repair, rug_confirmed, normal_repair_with/without_serial_no, return_receipt_location (with user domain), repair_location (readonly), job_location
  • Hidden state fields added after company_id: all 13 boolean stage-update flags (invisible=1)
  • Serial / product area: x_studio_serial_number (primary), complex attrs for lot_id visibility based on repair type
  • New notebook "Factory Repair Details" (visible only when job_location = "Factory Repair"): two groups for Sales Centre and Factory transfer dates
  • New notebook page "Repair Status" — Delivery Details group (driver name, vehicle, stage date) + Status Log with Created By/On 1-10 pairs
  • sale_order_id modified: domain filtered to partner, readonly when repair type set, invisible for "without serial no" type
  • Stage statusbar: clickable set to False (users cannot manually click stages)

Computed Fields (with Python logic)

These fields contain embedded Python compute methods managed by Studio. They have side effects: several auto-advance the ticket stage when conditions are met.
x_studio_task_status — Task Status (most complex)

Monitors FSM tasks for completion and auto-advances stage + populates materials/pricing fields.

for record in self: tasks = record.fsm_task_ids if tasks: done = all(t.fsm_done or record.x_studio_end_quick_repair for t in tasks) # Check SO delivery state so_delivered = all(t.sale_order_id.x_studio_valid_delivered_so for t in tasks if t.sale_order_id) if done and so_delivered: # Populate items/qty/sales_price from SO line # Auto-advance stage to 9 or 28 record.write({'stage_id': 9, 'x_studio_items': ..., 'x_studio_qty': ...})
x_studio_handed_over — Handed Over

Monitors stock.picking completion; when all pickings done, marks ticket as handed over and advances stage to 13/32.

x_studio_valid_invoiced_so → stage 3/22

Checks FSM tasks' sale orders for invoice payment. When invoiced, auto-advances to the "Invoiced" stage.

x_studio_valid_confirmed_so → stage 10/29

Checks FSM tasks' x_studio_valid_confirm_so. When confirmed, advances to the corresponding confirmation stage.

x_studio_valid_confirmed2_so → stage 12/31

Secondary confirmation check — advances to stage 12/31 when x_studio_valid_confirm2_so is set on FSM tasks.

x_studio_valid_delivered_so → stage 9/28 or 11/30

Complex: checks both x_studio_valid_delivered_so and x_studio_valid_delivered_so2 on tasks. Routes to different stages based on which flag is set.

x_studio_fsm_task_done — FSM Task Done

Returns True when all FSM tasks are marked done (fsm_done) or quick repair is ended. Upstream trigger for x_studio_task_status.

x_studio_fully_paid_so — Fully Paid SO

Checks x_studio_fully_invoiced_so on FSM tasks' sale orders.

x_studio_rug_approval_status — RUG Approval Status

Reads from sale.order via FSM task; returns "Approved" / "Rejected" based on x_studio_rug_approved/x_studio_rug_rejected flags on the sale order.

x_studio_material_availability — Material Availability

Reads x_studio_material_availability from FSM task records.

x_studio_user_location_validation — User Location Validation

Validates that the return receipt location is accessible to the current user based on x_studio_users_stock_location on the stock.location model.

x_studio_re_estimate_count / x_studio_re_estimate_status

Reads re-estimate count from FSM task sale orders. When count > 0, sets status to "Re-estimated".

x_x_studio_created_from_help_ticket_stock_picking_count

search_count on stock.picking filtered by x_studio_created_from_help_ticket = this ticket ID. Powers the "Repair Trans." stat button.

10 · Observations & Notes for Module Export

Strengths

  • The repair workflow is fully self-contained within the Helpdesk module — no custom module required for core logic.
  • Computed field auto-stage advancement is a clean pattern; it avoids manual stage manipulation by users.
  • Non-clickable statusbar enforces process integrity — users cannot skip stages.
  • Repair type classification via ticket.type flags (RUG / normal-with-SN / normal-without-SN) is elegantly extensible — adding a new type only requires setting flags on the ticket type record.
  • Multi-company support on helpdesk.stage via x_studio_company_id.

Issues & Technical Debt

Duplicate / legacy fields:
  • x_studio_serial_number (Serial Number-11) and x_studio_serial_no (Serial Number) both exist — two separate serial fields on the same model. The form view removes the first field (replace with no replacement) after layering in the second, suggesting x_studio_serial_number is vestigial.
  • x_studio_source_location / x_studio_source_location_1 and x_studio_virtual_location / x_studio_virtual_location_1 — duplicate related fields pointing to slightly different user chains.
  • 10 pairs of x_studio_created_by_N / x_studio_created_on_N is a brittle pattern — a proper audit log would use a one2many child model.
Inactive automation rules with multiple versions: Four versions of "RR - Auto Select Product for RUG Repairs" exist (ids 172, 173, 243, 175). Only two are active. When exporting to a module, only the active rules should be included; the others should be cleaned up.
Commented-out button XML: The form-button view contains several large commented-out <!--xpath--> blocks representing previous iterations of button logic. These should be removed before module export — they are dead code that increases view load time.
Test / exploratory fields: x_studio_cccc, x_studio_cccc3, x_studio_related_field_FNjnC, x_studio_related_field_MiNdm, x_studio_related_field_QuqN1 appear to be Studio experiments. The "Slowness - Test" automation rule (inactive) is a debugging artefact. These should be removed.
Report templates are copies of copies: Customer Letter, Repair Final Notice, and Repair Final Notice - Scrappage are _copy_N chains from the Repair Receipt template (UUID 73c7b165). The actual QWeb template XML should be reviewed to confirm each report has been properly customized — sometimes Studio copies retain the parent's content.
